# Zerodha AI Trader — Codex Planning Pack

This repository-planning pack is for a **personal, experimental, autonomous trading system** with:

- **Google Compute Engine VM** as the always-on runtime.
- A **reserved static external IP** for broker order traffic.
- **Zerodha Kite Connect** as the market-data and order integration.
- **OpenAI API** only as a structured trade-decision layer over Kite-derived technical data.
- **No OpenAI web search in the MVP**.
- **Codex** used to implement one plan at a time.

## Intended MVP

The MVP is a controlled experiment with approximately **₹2,000 allocated capital**. It is not a profit promise or an investment recommendation system.

Initial live boundaries:

| Item | MVP rule |
|---|---|
| Trading scope | Personal Zerodha account only |
| Live direction | Long-only cash-equity experiment |
| Product | Intraday-only configuration selected explicitly before live mode |
| Live strategies | One enabled initially; others remain in PAPER mode |
| Maximum open positions | 1 |
| Starting max trade notional | ₹500 |
| Starting max planned risk per trade | ₹10 |
| Starting max trades per day | 1 |
| Starting max daily loss | ₹20 |
| Order entry | Limit orders only initially |
| Protective exit | Must be placed/confirmed immediately after an entry fill |
| AI permissions | May approve/veto a structured candidate; never sizes or directly places an order |
| Default deployment state | `TRADING_MODE=OFF`, `LIVE_ARMED=false` |

The risk limits are intentionally tiny because the first objective is to validate system behaviour, broker integration, logs, stop handling and emergency controls.

## Why Kite Connect, not MCP, is the runtime core

The autonomous system needs deterministic backend control of data, orders, sessions, risk checks and journaling. Use Kite Connect REST/WebSocket APIs directly from the backend. An MCP connection may later be added as a read-only operator convenience, but it must not become a bypass around the risk engine or execution gateway.

## What Kite data can and cannot do

The MVP intentionally avoids OpenAI web search. Kite data can supply market quotes, streaming price/depth information, order updates, account/margin information and historical candles according to the API subscription and entitlements available to your app.

Kite data is **not** a replacement for company-news or corporate-event research. For the MVP, do not let the agent make claims about news, earnings, announcements or fundamentals. It decides only from structured technical/context data generated by your backend.

## How to use this pack with Codex

1. Create a new repository named `zerodha-ai-trader`.
2. Copy all files from this ZIP into the repository root.
3. Read `AGENTS.md`, `docs/ARCHITECTURE.md` and `docs/RISK_RULES.md` yourself.
4. Start Codex with `docs/plans/P00_PRODUCT_SCOPE_AND_PREREQUISITES.md`.
5. Execute only one plan per Codex session or branch.
6. Review tests and diffs before moving to the next plan.
7. Never arm live mode until all paper/shadow and kill-switch gates in `P12_MICRO_LIVE_ROLLOUT.md` pass.

## Non-negotiable operational facts

- Kite Connect authentication uses a backend token exchange; never put `api_secret` in frontend/mobile code.
- Access tokens expire at 6 AM the following day unless invalidated earlier, so an autonomous trading day still starts with a deliberate authenticated session setup.
- Live WebSocket health and order updates must be monitored.
- Broker order acceptance does not equal an executed trade; persist broker order-state events and fills.
- The order-placement service must use the configured static egress IP required by broker/API policy.
- There is no assumption of a broker sandbox; live connectivity tests can have real consequences.
